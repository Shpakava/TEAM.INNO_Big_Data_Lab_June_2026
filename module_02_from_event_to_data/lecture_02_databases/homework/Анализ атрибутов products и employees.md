Для сущностей products (товары) и employees (сотрудники) в системе хранятся важные бизнес-метрики и персональные данные.

Сущность products (Товары):
1. price (Цена): Числовой атрибут, определяющий базовую стоимость единицы товара без учета скидок. Это показатель, на основе которого рассчитывается выручка (total_price в таблице продаж).
2. vitality_days (Срок годности в днях): Специфический для продуктового ритейла атрибут. Он указывает, в течение скольких дней товар остается свежим. Позволяет системе автоматически сигнализировать о необходимости списания просрочки.
3. is_allergic (Наличие аллергенов): Логический флаг, указывающий, содержит ли продукт опасные для аллергиков компоненты. Критически важен для маркировки товаров и информирования покупателей.
4. category_id (Категория): Внешний ключ (FK), связывающий товар с categories. Позволяет группировать товары при построении отчетов.

Сущность employees (Сотрудники):
1. shop_id (Идентификатор магазина): Внешний ключ (FK), привязывающий сотрудника к конкретному магазину сети (shops).
2. hire_date (Дата найма): Дата, фиксирующая официальное начало работы сотрудника в компании. Используется HR-отделом для расчета стажа работы, начисления бонусов за выслугу лет и анализа текучести кадров.
3. birth_date (Дата рождения): Атрибут типа "дата", необходимый для контроля возраста сотрудников (например, соблюдение трудового законодательства в отношении несовершеннолетних) и для внутренних корпоративных процессов (поздравления, расчет среднего возраста коллектива).
4. first_name / last_name (Имя и Фамилия): Текстовые персональные данные. Например, для вывода имени кассира на печатный чек.

В теории реляционных баз данных атомарность означает, что значение в каждой ячейке таблицы должно быть неделимым (простым). В ячейке не может храниться массив, список, несколько значений через запятую или повторяющаяся группа данных.

Сущность products (Товары)
Список всех атрибутов (столбцов) и объяснение атомарности каждого атрибута:
1. product_id (Первичный ключ): Каждому товару присваивается ровно один уникальный идентификатор. Товар не может иметь список из нескольких ID в одной строке.
2. product_name: Содержит одну конкретную строку с названием товара. Оно неделимо: не нужно дробить название на отдельные буквы или слоги, оно представляет собой единый смысловой объект.
3. price: В каждый конкретный момент времени товар имеет ровно одну базовую цену. Ячейка не содержит историю цен или список цен для разных категорий покупателей в виде массива.
4. category_id: Внешний ключ, указывающий на конкретную категорию из таблицы categories. Один товар принадлежит ровно одной основной категории. Здесь нет списка категорий в одной ячейке.
5. class: Товар относится к одному конкретному классу без перечисления нескольких вариантов в одной ячейке.
6. modify_timestamp: Хранит одну конкретную точку во времени. Это неделимое значение системного таймстемпа.
7. resistant: Одиночный признак, хранит одно конкретное значение характеристики устойчивости.
8. is_allergic: Логический тип (Boolean), который может принимать только одно из двух значений: Yes (содержит аллергены) или No (не содержит). Нельзя записать оба значения одновременно.
9. vitality_days: Одно целое число, выражающее срок годности. Оно обозначает количество дней и не может содержать диапазон или список чисел в одной ячейке.

Сущность employees (Сотрудники)
Список всех атрибутов (столбцов) и объяснение атомарности каждого атрибута:
1. employee_id (Первичный ключ): Уникальный номер сотрудника. У одного человека не может быть несколько табельных номеров в одной ячейке.
2.  Разделение имени (first_name, middle_initial, last_name): 
    Разделение ФИО на 3 разных столбца — это классический пример приведения данных к атомарному виду. Вместо хранения одной строки данные разбиты на 3 неделимые части. 
    *   first_name содержит только одно имя.
    *   middle_initial содержит ровно одну букву (инициал).
    *   last_name содержит только одну фамилию.
3. birth_date: Ячейка содержит ровно одну конкретную дату.
4. gender: Код пола. Сотрудник имеет один конкретный гендерный маркер в системе.
5. city_id: Внешний ключ, указывающий на город проживания сотрудника. Предполагается, что сотрудник живет в одном конкретном городе.
6. shop_id: Внешний ключ, привязывающий сотрудника к конкретному магазину. Сотрудник официально числится и работает в одном конкретном филиале.
7. hire_date: Одна конкретная дата официального приема на работу.
